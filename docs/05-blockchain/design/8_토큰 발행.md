_GreatDIY 프로토콜의 가치 축적(Value Accrual)·유틸리티 중심 ERC20 토큰_

---

# **1. 설계 방향 요약**

GreatDIY 경제 구조는 아래 원칙을 바탕으로 설계된다:

1. **ERC20 토큰은 “1개만” 존재한다.**
2. **토큰 자체는 순수 ERC20 + burn 기능만 가지며, 어떠한 전송 제한·락업 로직도 내장하지 않는다.**
3. **투자자별 락업·베스팅·중도 철회는 모두 “VestingVault”에서 처리한다.**
4. **revocation(중도 철회) 기능은 필수적으로 지원한다.**
    - 단, 각 스케줄마다 revocable 여부를 개별 지정
5. 토큰 공급 총량은 고정되고, 추가 발행은 없다.
6. Buyback/Burn 흐름은 별도 BuybackTreasury 컨트랙트에서 수행한다.

이 구조가 **투자자 신뢰성·DEX 상장 용이성·보안성·운영성** 모두에서 최적이다.

---

# **2. G8DToken — 토큰 자체의 구조**

### **2-1. 핵심 기능**

- ERC20 (OpenZeppelin 기반)
- Burnable (직접 소각 기능)
- Mint 없음 (총량 고정)
- Pause / Blacklist / Transfer 제한 기능 없음
- Owner는 초기 배포 후 소각하거나, 최소 권한만 유지

### **2-2. 이유**

- 토큰 자체에 락업·통제 로직을 넣으면  
    ① 투자자들에게 “중앙화된 통제 위험”으로 보이고  
    ② CEX 상장, 유동성 공급, DEX 상호작용에서 복잡성이 생기기 때문.

→ 가장 최적의 구조는 **“순수 ERC20 + burn만 지원하는 불변 토큰”**.

---

# **3. 초기 발행 구조**

### **3-1. 총 공급량**

- 예: `1,000,000,000 G8D`
- 배포 시 전량을 **Founder(너)의 Monad 지갑**으로 발행
- 이후 VestingVault로 필요한 만큼 이동

### **3-2. 배포 시점 단계**

1. **G8DToken 배포**  
    → 1B G8D가 Liza 지갑으로 들어옴
2. **Liza 지갑에서 카테고리별로 분배**
    - 즉시 유통 물량(마케팅·유동성): 지갑/DEX
    - 락업 물량(투자자·팀·컨트리뷰터): VestingVault
3. **Vault로 이동한 물량은 Vesting 스케줄에 따라 시간 경과에 따라 claim 가능**
4. **BuybackTreasury 및 FeeDistributor가 이후 소각을 처리**

이 구조로 **초기 분배 + 락업 + 운영까지 모든 로직이 깔끔해진다.**

---

# **4. VestingVault — 락업·베스팅·철회 로직 담당**

### **4-1. 왜 필수인가?**

- **EOA(일반 지갑)에 토큰을 보내면 절대 회수 불가.**
- 투자자별로 조건이 다르므로  
    “컨트랙트 여러 개”가 아니라  
    **컨트랙트는 1개 / 사용자마다 서로 다른 Schedule**이 저장되는 구조가 가장 효율적.

### **4-2. Schedule 구조**

`struct Schedule {     uint256 total;         // 전체 할당량     uint256 released;      // 이미 투자자가 받은 수량     uint64  start;         // 베스팅 시작 시점     uint64  cliff;         // 클리프(언락 시작 시점)     uint64  duration;       // 전체 베스팅 기간     bool    revocable;      // 철회 가능 여부     bool    revoked;        // 철회 여부 }`

### **4-3. 기능 요약**

- `depositFor(beneficiary, amount, cliff, duration, revocable)`
- `claim()` — 현재까지 vesting된 물량만큼 수령
- `revoke()` — revocable일 경우
    - 지금까지 vested된 부분만 투자자가 받을 수 있고
    - vesting되지 않은 잔여 토큰은 프로젝트로 회수됨
- 투자자 스스로 중지(`cancelMyVesting`) 옵션도 확장 가능

### **4-4. 장점**

- 투자자별로 완전히 다른 조건 설정 가능
- 토큰 자체를 건드리지 않아 투자자 신뢰도 증가
- 회수/재배분/조건 변경이 명확하고 투명함
- 온체인 상태가 명확해 문서·프론트 UI 표시도 용이

# **5. Revocation (중도 철회) — 최종 선택**

### ✔ 반드시 지원

- 기여자/파트너/팀이 중도 이탈 시 락업 지속할 이유 없음
- 미베스트 물량은 회수해야 원칙적

### ✔ 투자자마다 revocable 여부를 개별 설정

- 기관 투자자: 보통 revocable=false
- 팀/파트너/컨트리뷰터: revocable=true
- 일부 전략적 투자자: revocable=true 또는 false, 계약에 따라

### ✔ revoke 시 처리

- vested(이미 시간 경과에 따라 언락된 부분)만 투자자 몫
- 나머지는 모두 회수 후 Treasury로 다시 이동

이 구조는 **전 세계 토큰 프로젝트들이 가장 널리 채택하는 안전한 패턴**이다.

---

# **6. Buyback/Burn과의 연계**

- FeeDistributor → BuybackTreasury → DEX → G8D 매수 → burn
- G8DToken은 `burn()`만 지원하면 충분
- VestingVault와 BuybackTreasury는 서로 독립적인 모듈로 유지

이렇게 하면 **시장 매출 → 토큰 가치 상승 → Supply 감소** 구조 완성.

---

# **7. 어떤 경우에도 하지 말아야 할 것**

### ❌ 투자자 지갑에 락업 조건을 전송 후 “나중에 막는 것”

→ 불가능 + 보안 위험 + 신뢰도 하락

### ❌ 투자자마다 ERC20 토큰을 따로 발행

→ 가격·유동성·상장·인식 모두 파탄

### ❌ 토큰 컨트랙트에 전송 제한·입출금 잠금 기능을 넣는 것

→ 중앙화 논란 + CEX 상장 불가능에 가까움

### ❌ 업그레이드 가능한 Proxy 토큰

→ 큰 신뢰 리스크 + 해킹 위험도 증가

**→ 지금 선택한 구조가 Best Practice이자 가장 안전한 길이다.**

# **8. 최종 모델 요약 (아래 내용을 그대로 Token Issuance 문서 핵심 요약으로 사용)**

### **토큰(G8DToken)**

- MONAD L1 기반 ERC20
- burn 가능
- 총량 고정 / 추가 발행 없음
- 전송 제한 없음 (pure ERC20)

### **초기 발행 구조**

- 배포 시 총량 전부 Founder 지갑으로
- 락업 물량은 VestingVault로 이동
- 즉시 유통되는 물량은 직접 전송 또는 DEX 유동성 공급

### **락업·베스팅 구조 (VestingVault)**

- 컨트랙트는 1종류
- 투자자마다 다른 스케줄을 레코드로 저장
- revocable 지원
- revoke 시 vested만 지급, 나머지는 회수
- 팀·기여자 물량은 revocable=true
- 기관 등 주요 투자자는 revocable=false 가능 (계약 조건에 따라)

### **Buyback/Burn**

- FeeDistributor → BuybackTreasury → DEX → G8D 소각
- 토큰 공급량 점진적 감소 (Deflationary Model)
